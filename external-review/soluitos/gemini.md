# **Exhaustive Analysis of the 8x Performance Degradation in Kotlin Compose Desktop GIO File Listing Pipelines**

The integration of native system libraries within modern Java Virtual Machine (JVM) frameworks has undergone a paradigm shift with the introduction of Project Panama and the Foreign Function & Memory (FFM) API. By replacing the legacy Java Native Interface (JNI), FFM promises near-native execution speeds for Kotlin and Java applications interacting with C libraries. However, when interfacing complex asynchronous systems—such as the GNOME GLib/GIO virtual file system—with the Kotlin Coroutine concurrency model and the Jetpack Compose Desktop rendering engine, severe performance anomalies can emerge.  
This report provides a comprehensive, exhaustive deconstruction of a critical performance degradation observed in a Kotlin Compose Desktop file manager application. The anomaly manifests as an 8x throughput collapse when comparing an isolated headless benchmark against the production application running the Compose graphical user interface. While the benchmark processes a directory of 440 items in 121 milliseconds, the exact same backend code path executing within the production UI pipeline requires approximately 1000 milliseconds.  
Through a rigorous examination of the application's source code architecture, the GNOME GLib GMainContext event loop mechanics, the Java FFM native bridging implementation, and the Kotlin Coroutine dispatch models, this analysis identifies the exact mechanisms responsible for this performance collapse. The investigation isolates the bottleneck not within the GNOME I/O operations themselves, but within a catastrophic intersection of native mutex lock contention, Kotlin coroutine thread starvation, and Foreign Function boundary crossing overhead induced specifically by the active UI environment.

## **The Architectural Anatomy of the File Listing Pipeline**

To isolate the precise locus of the performance degradation, it is first necessary to deconstruct the exact execution sequence, data structures, and asynchronous flow control mechanisms that constitute the application's file listing architecture. The pipeline is engineered as a highly concurrent producer-consumer system that relies on a complex interaction between Kotlin Coroutines, Java FFM native upcalls, and the native GLib event loop.

### **State Management and Trigger Mechanisms**

The data retrieval sequence is initiated and managed within the directory state management module. The system guarantees state safety and prevents interleaved data corruption by rigorously canceling previous coroutine jobs and resetting StateFlow structures prior to initiating a new listing operation.1 The initialization function invokes a dedicated listing strategy, which subsequently delegates the native retrieval to the underlying GNOME I/O backend.1  
The resulting stream of file information is handled via a Kotlin Flow. To optimize the user experience and ensure that the Compose UI remains highly responsive, the pipeline employs a chunking operator that progressively batches the incoming items. The collection mechanism is configured to emit an initial small batch of 75 items to facilitate an immediate "time-to-first-render," followed by subsequent larger batches of 200 items.1  
Upon receiving a chunk of file metadata, the state manager atomically updates its internal associative map using thread-safe primitives. This updated mapping is immediately converted to a standard immutable list and propagated to the underlying StateFlow structure. This propagation acts as the direct trigger for the Compose Desktop engine to invalidate its current state and execute a recomposition pass on the LazyVerticalGrid displaying the files.1

### **The Asynchronous Data Producer**

The core data retrieval engine utilizes a channelFlow builder to construct a concurrent pipeline capable of overlapping native I/O operations with JVM object instantiation.1 The flow initiates a native GNOME I/O asynchronous directory enumeration operation through the Java FFM bindings.  
To maximize throughput and minimize the overhead of native transitions, the backend implements a progressive batching loop. Rather than fetching file pointers sequentially, the engine requests native file pointers in batch sizes that scale dynamically from 75 to 100, then 300, and finally cap at 500 files per native request.1  
As soon as a batch of native file pointers is returned from the GNOME I/O library, the backend dispatches the CPU-heavy parsing and FFM mapping workload to background worker threads. This dispatch utilizes Dispatchers.IO, a bounded thread pool designed for offloading blocking operations.1 To prevent unbounded thread proliferation and manage JVM memory pressure, the backend enforces a strict concurrency limit using a semaphore initialized with eight permits. This restricts the system to a maximum of eight concurrent worker coroutines performing FFM mappings simultaneously.1 After the native pointers are mapped to Kotlin data classes, the items are dispatched downstream via a thread-safe channel mechanism, buffered to handle temporary backpressure, and transferred to the UI collector.1

### **Foreign Function and Memory Mapping Overhead**

Within the boundaries of the permitted background workers, each native GNOME file pointer must be translated into a Kotlin domain model. This translation occurs through a dedicated type mapping module. Historically, invoking native C code from the JVM via FFM incurs overhead due to the necessity of transitioning execution contexts, pinning memory, and managing garbage collection boundaries.1  
The pipeline implements an aggressive optimization pathway designated as the minimal listing mode. This mode is specifically engineered to bypass deep, expensive FFM calls when enumerating massive directories. To eliminate cryptographic overhead, the mapper assigns a static sentinel UUID rather than generating a randomized identifier for each listed file. Furthermore, it bypasses complex and synchronous permission lookups, noting that skipping executable checks alone saves at least one FFM native transition per file.1 The mapper also implements a fast-path string concatenation algorithm for child URI computation, bypassing native GNOME child resolution calls whenever standard alphanumeric characters are detected.1  
Despite these extensive optimizations, the mapping module must still execute multiple FFM transitions per file to extract essential metadata. The system must natively query the file size, read the content type string to determine MIME classifications, evaluate native boolean flags for hidden status, and extract native modification timestamps.1 Each of these operations represents a discrete transition across the Java FFM boundary.

### **The Coroutine-to-Native Execution Bridge**

The most complex and fragile intersection of the entire architecture is the bridging mechanism that translates GNOME's asynchronous C-callbacks into standard Kotlin suspending functions. The GNOME I/O library operates on a callback-driven asynchronous model, requiring native function pointers to trigger upon operation completion.  
The bridging module wraps these native C-operations within a Kotlin suspendable coroutine. The implementation must carefully navigate the volatile nature of JVM memory management. Because the Java FFM wrappers encapsulating the native callback logic are managed by the JVM's garbage collector, there exists a profound risk of premature reclamation. The FFM API utilizes background cleaner actions to free native memory segments when the corresponding Java objects become unreachable.3 If the JVM garbage collector reclaims the callback object while the native GNOME library is still executing the asynchronous operation, the native upcall stub is destroyed. When the GNOME library eventually attempts to invoke the callback, it will trigger a fatal segmentation fault within the native process.1  
To circumvent this architectural hazard, the bridging module explicitly pins the native callback wrapper by assigning it to a strong reference declared outside the scope of the suspension block. Because the Kotlin Coroutine state machine maintains a strong reference to this variable while the coroutine remains suspended, the JVM garbage collector is prevented from reclaiming the FFM upcall stub.1  
To drive these native callbacks, the application relies on the GNOME GLib event loop. To ensure that callbacks execute with minimal latency, the architecture spawns a dedicated daemon operating system thread. This thread continuously executes blocking iterations on the default GLib main context, native blocking the OS thread until I/O events or timeouts arrive, thereby explicitly bypassing the polling latencies typically associated with integrating external event loops into a UI framework.1

## **Diagnostic Deconstruction of the 8x Degradation**

The empirical timing logs provided within the system diagnostics exhibit a catastrophic and isolated slowdown specifically within the worker processing phase of the pipeline. In the headless benchmark environment, the application processes files at a rate of approximately 0.27 milliseconds per file, culminating in a total execution time of 121 milliseconds for 440 items. Conversely, the production pipeline requires approximately 1.9 milliseconds per file, extending the total execution time to roughly 1000 milliseconds.  
Because both execution environments utilize the exact same backend logic, identical chunking algorithms, and the precise same FFM call pathways, the degradation cannot be attributed to algorithmic inefficiency, poorly optimized data structures, or fundamental flaws in the Kotlin language's execution speed.  
An analysis of the application's timing telemetry reveals a crucial insight into the nature of the bottleneck. The telemetry logs indicate that the initial GNOME asynchronous directory enumeration completes in a mere 10 milliseconds in both environments. Furthermore, the log entry gio\_list\_done=+153ms demonstrates that the GNOME asynchronous fetch operations are completing rapidly. By the time the 153-millisecond mark is reached, the GNOME library has already fetched the native file pointers, but the Kotlin background workers have only managed to process and emit 28 items.  
This telemetry proves definitively that the bottleneck does not reside within the native file system, the solid-state drive I/O, or the GNOME C-library's fetching algorithms. The bottleneck is entirely localized within the eight background worker coroutines attempting to map the native memory structures to Kotlin objects.  
Furthermore, extensive developer testing has systematically eliminated several common Kotlin Flow and UI rendering bottlenecks:  
The application of unlimited channel buffering yielded no improvement, explicitly ruling out downstream backpressure from the Compose UI as the primary limiting factor. The producer threads are being actively starved or natively blocked, rather than waiting for the consumer to process the stream. Modifying the flow operator mechanics by removing the chunking protocol degraded performance further due to the introduction of per-item cooperative yielding overhead. Modifying the internal state management to utilize mutable accumulators rather than immutable state copies slowed the system by hundreds of milliseconds, validating that the Kotlin StateFlow equality checks and immutable reference updates are operating at peak efficiency.  
The performance gap is also highly variable, fluctuating between 986 milliseconds and 1721 milliseconds across identical runs. This extreme variance, coupled with the isolated worker slowdown and the elimination of flow control issues, points unequivocally to a systemic conflict involving resource contention, thread starvation, or native mutex lock contention introduced exclusively when the Compose Desktop graphical user interface environment is active.

## **Root Cause Analysis: The Confluence of Compose, FFM, and GLib**

The 8x performance degradation is not a single bug, but rather the result of a compounding architectural conflict between how Kotlin Coroutines distribute concurrent workloads, how the Java Foreign Function and Memory API transitions execution contexts, and how the GNOME GLib library enforces thread safety within its native event loop. The production UI environment introduces a severe state of lock contention that the isolated headless benchmark naturally avoids.

### **The Dynamics of Native Lock Contention**

The GNOME GLib event loop architecture revolves entirely around the GMainContext data structure. A main context is a generalized implementation of an event loop, functioning fundamentally as a sophisticated polling mechanism utilized for tracking file descriptors, executing asynchronous callbacks, and managing event-based widget systems.5 The overarching design philosophy of GLib dictates that a specific GMainContext can only be actively running and iterating within a single thread at any given time, although event sources and asynchronous operations can be safely attached to it from secondary threads.7  
To protect the internal linked lists, file descriptor arrays, and priority queues of the main context from simultaneous memory corruption when accessed by multiple threads, GLib employs native C-level mutex locks. Whenever a thread interacts with the context—whether it is iterating the polling loop, adding a new event source, waking up the context, or dispatching an asynchronous callback—it must first acquire explicit ownership of the context via the g\_main\_context\_acquire native function.8 If another thread currently holds ownership of the context, the attempting thread is natively blocked until the mutex is released.10  
In the application's current architectural design, several competing entities are forcefully interacting with the global default GMainContext. First, the dedicated native pump daemon thread executes continuous, blocking iterations on the default context.1 This causes the pump thread to hold the context lock heavily and frequently while waiting for or dispatching native events. Second, the FFM pipeline utilizes eight parallel worker coroutines running on Dispatchers.IO. These eight worker threads continuously execute asynchronous GNOME FFM calls.  
When an asynchronous function is invoked from one of these eight worker threads, the GNOME library utilizes its internal GTask API to manage the asynchronous operation.13 The GTask specification explicitly requires that the completion callback must be invoked in the "thread-default" main context of the thread that initiated the operation.13 Because the Kotlin worker threads are standard JVM carrier threads that have not explicitly defined a custom thread-default context via g\_main\_context\_push\_thread\_default, the GLib library automatically falls back to utilizing the global default GMainContext.5  
The headless benchmark performs optimally because the system is highly isolated. In the benchmark environment, the only entities interacting with the global default context are the isolated pump thread and the eight Kotlin worker coroutines. While there is minor contention for the native mutex, the lack of external interference allows the FFM calls to proceed rapidly, achieving the 0.27 milliseconds per file processing rate.  
The production scenario introduces a catastrophic disruption to this equilibrium. When the Compose Desktop UI is instantiated, the execution environment changes radically. Modern Linux desktop environments, GTK integration layers, display servers, clipboard managers, and accessibility services are inherently intertwined with the native GLib and GTK infrastructure.  
The Compose main thread, its underlying render threads, and various system-level background processes introduce massive, uncoordinated traffic to the global default GMainContext. When the eight Kotlin worker threads attempt to execute their FFM calls, they must cross the native boundary and compete for the context lock against the native pump thread, the Compose UI thread, and the entire host operating system.19  
When a native thread lock is heavily contended, the calling JVM threads are natively blocked at the operating system level. The extreme increase in processing time—from 0.27 milliseconds to 1.9 milliseconds per file—is directly quantifiable as the wait time spent by the JVM threads suspended on the native GMainContext mutex.  
The following table models the lock contention environment across both execution states:

| Execution Environment | Compose UI Thread Active | Contending Entities on GMainContext Mutex | Kotlin Worker Thread State | Per-File Processing Time |
| :---- | :---- | :---- | :---- | :---- |
| Headless Benchmark | False | Pump Thread, 8 Kotlin Workers | Rarely Blocked (Low Contention) | 0.27 milliseconds |
| Production Pipeline | True | Pump Thread, 8 Kotlin Workers, UI Render Threads, OS Services | Severely Blocked (High Contention) | 1.90 milliseconds |

### **Thread Starvation within the Coroutine Dispatcher**

The native lock contention introduced by the GMainContext triggers a severe secondary failure cascade within the Kotlin Coroutine machinery. Kotlin Coroutines achieve concurrency by multiplexing thousands of lightweight suspension structures onto a small pool of heavy operating system threads, known as carrier threads.  
The application utilizes Dispatchers.IO for its background workers. This dispatcher is a bounded thread pool, dynamically scaling to a maximum of 64 threads or the total number of available CPU cores, whichever value is higher.2 While Kotlin coroutines are designed to suspend cooperatively—releasing the underlying carrier thread back to the pool when waiting for network data or delays—FFM calls into native C-libraries bypass this cooperative model.3  
From the perspective of the JVM, an FFM downcall is a strictly synchronous, blocking operation. When the native GMainContext lock blocks the execution of the GNOME FFM function, the underlying JVM carrier thread is physically pinned by the operating system. It cannot be decoupled, and it cannot execute other pending coroutines.3  
Because the pipeline utilizes a semaphore to allow eight concurrent FFM workers, eight threads within the Dispatchers.IO pool are immediately pinned and blocked waiting for the native mutex. In a robust desktop application, Jetpack Compose Desktop also heavily utilizes Dispatchers.IO and Dispatchers.Default for essential background tasks, including image decoding, asset loading, font resolution, and rendering calculations.2  
By pinning the IO threads with contended native FFM calls, the file listing pipeline actively starves the Compose UI of the CPU resources required to render the application. The telemetry data supports this starvation hypothesis entirely. The developers noted that attempting to decouple the collection by migrating it to Dispatchers.Default degraded the performance by an additional 262 milliseconds. Dispatchers.Default is strictly bounded to the exact CPU core count.2 Pinning a Dispatchers.Default carrier thread with a blocked FFM call instantly deprives the Compose UI of its primary calculation threads, leading to severe context-switching thrashing, CPU pipeline stalls, and cascading UI freezes.2

### **The Latency Penalty of Asynchronous Thread Hopping**

A third systemic inefficiency exacerbating the 8x degradation is found in the mechanism governing how the coroutine bridge handles the resumption of suspended execution states.  
When a GNOME asynchronous operation completes in the background, the native C-callback is executed by the specific thread that is currently owning and iterating the main context. In the current architecture, this is always the dedicated daemon pump thread.1  
Inside the native callback wrapper, the bridging code executes the resumption of the Kotlin continuation. Because this resumption is invoked directly from the native pump thread, the Kotlin Coroutine state machine must schedule the resumption of the suspended coroutine. The file listing flow is explicitly bound by a .flowOn(Dispatchers.IO) operator, mandating that the continuation must be dispatched back to the IO thread pool.1  
This architectural design creates a mandatory, high-latency cross-thread context switch for every single batch of files processed:

1. A thread on Dispatchers.IO initiates the asynchronous FFM call and suspends cooperatively.  
2. The GNOME C-library executes the physical file enumeration.  
3. The dedicated native pump thread triggers the C-callback upon completion.  
4. The native pump thread resumes the Kotlin continuation object.  
5. The Kotlin Dispatcher queues the workload back onto a thread within Dispatchers.IO to process the results.

In a highly contended production environment where the IO pool is already battling the Compose rendering engine for CPU time and dealing with carrier thread starvation, this context switch incurs immense scheduling latency.27 The overhead of waking up an IO thread, loading the JVM state machine, and pulling the native memory segments from the system RAM back into the CPU cache—which was likely just evicted by the Compose render loop processing the previous chunk of UI updates—adds substantial microsecond delays that accumulate linearly across hundreds of files.

### **FFM Memory Management and Garbage Collection Backpressure**

The introduction of Project Panama and the Java Foreign Function and Memory API provides a fundamentally different memory management paradigm compared to the legacy JNI model. FFM allocates contiguous blocks of native memory using Arena structures. In the application's current implementation, the underlying java-gi bridge utilizes an ArenaCloseAction.CLEANER protocol to ensure that native C-structures are safely freed when their corresponding Java wrapper objects are garbage collected.3  
In the isolated headless benchmark environment, memory allocation and garbage collection patterns are highly predictable and uniform. The background threads allocate FFM wrappers, process them, and release references in a steady, un-interrupted stream. The JVM garbage collector can efficiently process the cleaner queues and free the native memory without causing pauses.  
However, in the Compose production environment, the dynamic rendering engine generates a massive volume of short-lived objects during the LazyVerticalGrid recomposition phases. Every time a new chunk of files is emitted by the StateFlow, Compose allocates new node structures, layout modifiers, and text measurement models. This constant allocation triggers frequent minor garbage collection pauses.  
Because the background worker coroutines are generating thousands of temporary FFM memory wrappers per directory refresh, the JVM garbage collector must continuously process the background cleaner queues to free the underlying native C-structs. When the Compose UI is heavily allocating memory on the main threads, the background garbage collection threads must prioritize standard heap cleanup to prevent out-of-memory errors. This prioritization delays the execution of the cleaner queues. This complex interplay creates subtle back-pressure on the native memory allocators, incrementally slowing down the instantiation speeds for new FFM objects within the background worker threads.

## **Strategic Engineering Recommendations**

To permanently eradicate the 8x performance gap and restore benchmark-level execution speeds within the production Compose UI environment, the system architecture must be restructured to completely decouple the file listing FFM pipeline from the global GNOME GLib state. By eliminating native lock contention, preventing carrier thread starvation, and optimizing FFM data extraction, the application can achieve seamless desktop-grade rendering.  
The following engineering modifications provide a comprehensive roadmap for resolving the architectural bottleneck.

### **Implementation of Isolated Thread-Default Main Contexts**

The most critical and impactful architectural revision requires isolating the FFM asynchronous callbacks from the global default GMainContext utilized by the application pump thread and the Compose UI layer. The GNOME GLib library explicitly provides the capability to push a specialized "thread-default" main context.5  
When g\_main\_context\_push\_thread\_default() is invoked on a specific thread, any subsequent GNOME I/O asynchronous operations executed on that thread will automatically attach their event sources and dispatch their callbacks to this isolated context, completely bypassing the heavily contended global context.5  
The architecture should be refactored to eliminate the reliance on a global pump thread for file system operations. Instead, the application should instantiate a dedicated Kotlin Coroutine Dispatcher backed by a single, permanent native thread that exclusively owns its own GMainContext.

1. Instantiate a new, isolated GMainContext via the FFM bindings.  
2. Spawn a single, dedicated JVM thread specifically designed for GNOME I/O processing.  
3. Within the execution scope of this dedicated thread, utilize the g\_main\_context\_pusher\_new function or explicitly call g\_main\_context\_push\_thread\_default to set the isolated context as the thread default.17  
4. Wrap this dedicated thread within a custom Kotlin CoroutineDispatcher implementation.  
5. Execute the file enumeration channelFlow and all GNOME FFM callbacks entirely upon this isolated dispatcher.

By compartmentalizing the event loop, the file listing pipeline will never be forced to compete for the native LOCK\_CONTEXT mutex against the Compose UI rendering threads or the host operating system's background processes.19 The FFM boundary crossings will execute instantaneously, precisely mimicking the uncontended execution state observed in the headless benchmark.

### **Restructuring the Concurrent Worker Pipeline**

The current backend design utilizes a Semaphore(8) throttling mechanism to launch up to eight concurrent Dispatchers.IO worker coroutines to map FFM objects.1 While parallelization is generally advantageous for pure CPU-bound algorithms, it is highly counterproductive when interfacing with single-threaded native C-libraries via Panama FFM.  
Because FFM property getters must eventually cross the native boundary to access memory segments, parallelizing them across eight separate JVM carrier threads forces the JVM to synchronize multiple threads against the native memory segments and exacerbates native lock contention. Furthermore, it generates unnecessary context switching on the IO pool, which Compose relies upon for rendering.  
The Semaphore(8) structure and the parallel worker coroutine launches should be entirely eliminated. If the file mapping is executing within the isolated, single-threaded dispatcher context recommended above, the system should perform the FFM mapping sequentially within the batch retrieval loop.  
Sequential mapping on a single dedicated thread utilizing the Java 22+ Panama FFM API is heavily optimized by the JVM's Just-In-Time (JIT) compiler. The JIT compiler is capable of aggressively inlining the native downcall method handles, functionally eliminating the FFI transition overhead for repeated operations.4 Distributing the workload across eight parallel threads prevents the JIT compiler from optimizing a single hot execution path, destroys CPU cache locality, and inflates the allocation of temporary FFM wrapper objects.

### **Optimizing Foreign Function Property Extractions**

Even when operating in the highly optimized listingMode \= true pathway, the mapping module performs several discrete native boundary crossings per file.1 To maximize performance, the frequency of these crossings must be minimized.  
The extraction of native strings is particularly expensive via FFM. Operations such as gioInfo.contentType?.toString() require the JVM to allocate native memory, copy UTF-8 byte arrays across the boundary, and instantiate immutable JVM String objects.1 Because the vast majority of files in a directory will share identical MIME types (e.g., standard folders, text files, or image formats), string conversion represents a massive duplication of effort.  
The mapping module should implement native pointer caching for frequently repeating string attributes. Rather than converting every GNOME content type to a Kotlin String, the system should read the native C-string pointer address directly via FFM. This raw memory address can be compared against an internal JVM hash map of known pointer addresses. If the pointer has been encountered previously, the system can instantly return the cached Kotlin String reference without executing a costly byte-array copy or instantiating a new JVM string object.  
Furthermore, conditional FFM evaluations, such as checking if a parent path represents a remote file system, must be rigorously cached at the parent directory level entirely.1 Evaluating remote URI substring matches on a per-child basis across thousands of items generates cumulative CPU overhead that contributes directly to the delayed UI rendering times.  
The following table categorizes the FFM attribute extraction optimizations:

| FFM Property Target | Current Extraction Mechanism | Recommended Optimization Strategy |
| :---- | :---- | :---- |
| File Name | gioInfo.name?.toString() | Accept inherent FFM overhead; names are highly unique. |
| MIME Content Type | gioInfo.contentType?.toString() | Implement native pointer caching to bypass UTF-8 byte copying and object allocation for repeating MIME types. |
| Remote Status | Evaluated per child | Evaluate once at the parent directory initialization phase and propagate to all mapped children. |

### **Refining StateFlow Chunking and Recomposition Backpressure**

While the application metrics indicate that the actual data manipulation for StateFlow updates adds only marginal millisecond overhead, the frequency of these state emissions dictates exactly how often the Jetpack Compose engine must halt its execution and perform a full recomposition pass on the user interface.1  
In the current state management implementation, the flow chunking operator is rigidly configured to fixed sizes: initialSize \= 75, size \= 200\.1 For a standard directory containing 440 items, this configuration triggers three distinct StateFlow emissions and forces three separate, highly demanding UI recomposition passes within a constrained time span of 1000 milliseconds.  
This rigid chunking mechanism should be replaced with a dynamic, time-based debouncing or scaling algorithm.  
The system should continue to emit the initial batch of 75 items immediately upon retrieval to guarantee a rapid "time-to-first-render," satisfying the user's immediate visual expectations. However, for all subsequent retrieved batches, the state manager should buffer the updates based on elapsed time rather than fixed item counts. For example, accumulating updates for a minimum threshold of 100 to 150 milliseconds before pushing the mutated list to \_itemsList.value will drastically reduce UI thrashing.  
This approach prevents Jetpack Compose from executing rapid, overlapping UI recalculations that spike CPU usage, trigger severe garbage collection pressure, and starve the background FFM workers of processing power precisely when they are attempting to map the final batches of files.

### **Alternative Architectural Pathway: Native Background Task Integration**

If completely isolating the GMainContext via custom Kotlin coroutine dispatchers proves too architecturally complex or fundamentally incompatible with the existing java-gi bridging design, the pipeline should be re-engineered to bypass JVM coroutine parallelization entirely. The application should instead utilize the GNOME GLib native background task runner: GTask.13  
The GTask architecture provides a robust, thread-safe mechanism to execute blocking or CPU-intensive operations natively in a background C-thread belonging to GLib's internal thread pool, while guaranteeing that the result callback is safely dispatched back to the original main context.5  
Instead of dragging individual native file pointers across the Java FFM boundary into Kotlin and executing the mapping logic across Kotlin dispatcher threads, the application could construct a single native C-function. This function would iterate the GNOME file enumerator natively, extract the required primitive data, and construct a contiguous array of C-structs representing the entire batch of files. The JVM would then perform a single FFM downcall to trigger this native enumeration, and a single upcall would return the entire packed memory segment to Kotlin.  
This architectural paradigm shift would collapse hundreds of individual, high-overhead FFM boundary crossings into a single efficient transition, circumventing the Kotlin JVM threading bottlenecks and completely insulating the file listing performance from the Jetpack Compose rendering environment.  
By systematically addressing the native mutex lock contention, eliminating the destructive parallelization of FFM workers, and optimizing the interaction between the GLib event loop and the Kotlin Coroutine dispatchers, the 8x performance degradation will be completely resolved, ensuring that the Kotlin Compose Desktop application operates with uncompromising, native-grade efficiency.
